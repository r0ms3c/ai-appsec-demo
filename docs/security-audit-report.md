An unauthenticated attacker can, in a **single HTTP request**, obtain valid admin JWTs from `GET /admin/debug/tokens` — or forge one using the `alg=none` bypass — then chain SQL injection to exfiltrate the full database (including AWS credentials), path traversal to read arbitrary server files, and SSRF to pivot into internal infrastructure. The application has no compensating controls (rate limiting, audit logging, WAF) that would slow any of these chains.

## OWASP Top 10 Coverage

### A01 — Broken Access Control
🔴 **Confirmed — Multiple critical instances**

- `admin.py:28` — IDOR: no ownership check on `GET /admin/user/{user_id}`; any user retrieves any user's password hash
- `admin.py:73` — Mass assignment: users can set their own `role` to `admin` via `PUT /admin/user/{id}`
- `admin.py:119` — Broken function-level auth: `GET /admin/debug/tokens` requires zero authentication and returns valid signed JWTs for every user
- `db.py:131` — No ownership check on fund transfer: any user drains any account
- `files.py:34` — Path traversal on download: `../../etc/passwd` style attacks
- `files.py:60` — Path traversal on upload: attacker-controlled filename written directly to filesystem
- `files.py:92` — Directory listing: all users see all uploaded files from all other users
- `files.py:113` — No ownership check on delete: any user deletes any file

### A02 — Cryptographic Failures
🔴 **Confirmed — Pervasive**

- `auth.py:22` — JWT secret is literally `"secret"` — trivially brute-forceable
- `auth.py:28-43` — Passwords hashed with unsalted MD5; rainbow tables instantly crack all accounts
- `main.py:35-37` — Three hardcoded secrets: `SECRET_KEY`, `DB_PASSWORD`, `INTERNAL_API_KEY`
- `db.py:24-28` — DB credentials hardcoded in source
- `admin.py:99` — Hardcoded `ADMIN_KEY` in source control
- `main.py:65` — `DB_PASSWORD` actively returned in every HTTP 500 response body

### A03 — Injection
🔴 **Confirmed — Every database query is vulnerable**

- `db.py:87` — f-string SQL injection in `/data/account`
- `db.py:113` — f-string SQL injection in `/data/search` LIKE clause
- `db.py:138-151` — Three separate f-string injections in `/data/transfer` (from_account, to_account, note)
- `files.py:60-84` — Unrestricted file upload: no extension allowlist, no size cap, no content validation

### A05 — Security Misconfiguration
🔴 **Confirmed**

- `main.py:43` — `debug=True` enables verbose tracebacks
- `main.py:55-67` — Global 500 handler returns full Python traceback, `os.getcwd()`, and `DB_PASSWORD`
- `main.py:93-101` — `/health` returns `SECRET_KEY` and `dict(os.environ)` to any caller
- `db.py:93-95` — Raw SQL query string returned in HTTP error bodies
- `main.py:112` — Server binds to `0.0.0.0`

### A07 — Auth and Session Failures
🔴 **Confirmed**

- `auth.py:71` — Rate limiting is dead code: threshold `> 1000` and body is `pass`
- `auth.py:87-91` — Tokens have no `exp` claim; valid forever
- `auth.py:74-77` — Timing attack enables username enumeration
- `auth.py:126-141` — Password reset token is `MD5(username + floor(unix_ts/60))` — predictable, algorithm hint returned in response
- `auth.py:137` — Reset token returned directly in response body

### A08 — Software and Data Integrity Failures
🔴 **Confirmed**

- `auth.py:110-114` — `jwt.decode(..., algorithms=None)` accepts any algorithm including `"none"`, bypassing signature verification entirely
- `auth.py:114` — `options={"verify_exp": False}` disables expiry validation unconditionally

### A09 — Logging and Monitoring Failures
⚠️ **Confirmed** — No access logging, no failed-login alerting, no audit trail on fund transfers, user deletion, or admin panel access.

### A10 — SSRF
🔴 **Confirmed**

## Remediation Snippets

### Fix: JWT Algorithm Pinning

```python
# Before (vulnerable) — auth.py:110
payload = jwt.decode(token, JWT_SECRET, algorithms=None, options={"verify_exp": False})

# After (secure)
payload = jwt.decode(token, JWT_SECRET, algorithms=["HS256"])
```
**Why:** `algorithms=None` lets the attacker declare `alg=none` in the token header, bypassing HMAC verification. Pinning to `["HS256"]` server-side makes the header claim irrelevant.
**CWE:** CWE-345

---

### Fix: SQL Injection

```python
# Before (vulnerable) — db.py:87
query = f"SELECT * FROM accounts WHERE owner = '{owner}'"
result = db.execute(query).fetchall()

# After (secure)
result = db.execute("SELECT * FROM accounts WHERE owner = ?", (owner,)).fetchall()
```
**Why:** Parameterized queries send data and SQL structure separately — user input is never interpreted as SQL syntax.
**CWE:** CWE-89

---

### Fix: Path Traversal

```python
# Before (vulnerable) — files.py:41
file_path = os.path.join(UPLOAD_DIR, filename)

# After (secure)
from pathlib import Path
safe_path = (Path(UPLOAD_DIR).resolve() / Path(filename).name).resolve()
if not str(safe_path).startswith(str(Path(UPLOAD_DIR).resolve())):
    raise HTTPException(status_code=400, detail="Invalid filename")
file_path = safe_path
```
**Why:** `Path.name` strips directory components; `.resolve()` canonicalizes `..` segments; the prefix check ensures the result stays inside the allowed directory.
**CWE:** CWE-22

---

### Fix: Password Hashing

```python
# Before (vulnerable) — auth.py:81
password_hash = hashlib.md5(password.encode()).hexdigest()

# After (secure)
import bcrypt
password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()

# Verification
def verify_password(plain: str, stored_hash: str) -> bool:
    return bcrypt.checkpw(plain.encode(), stored_hash.encode())
```
**Why:** MD5 has no work factor — a GPU cracks billions per second. bcrypt includes a per-password salt and a tunable cost factor designed specifically for password storage.
**CWE:** CWE-916

### Fix: Error Handler

```python
# Before (vulnerable) — main.py:55-67
return JSONResponse(status_code=500, content={
    "error": str(exc),
    "traceback": traceback.format_exc(),
    "internal_path": os.getcwd(),
    "db_password": DB_PASSWORD,
})

# After (secure)
import logging
logger = logging.getLogger(__name__)

@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    logger.exception("Unhandled error on %s %s", request.method, request.url.path)
    return JSONResponse(status_code=500, content={"error": "Internal server error"})
```
**Why:** Stack traces, working directories, and database passwords have no place in HTTP responses. Log details server-side for debugging; return a generic message to clients.
**CWE:** CWE-209
