## What this is

Most engineers use AI coding assistants like autocomplete with extra steps. This project demonstrates a different approach: **Claude Code as a security engineering system** вЂ” with domain-specific agents, enforced policies, and automated workflows wired into CI/CD.

Built around a deliberately vulnerable Python API (VulnBank), the platform shows Claude Code finding real vulnerabilities, generating threat models, producing Sigma detection rules, and enforcing security policy architecturally via hooks вЂ” with all output documented as real reports.

## How it works

Claude Code has four extension primitives. Using all four correctly вЂ” each for its intended purpose вЂ” is where the productivity gains come from.

| Primitive | Layer | Security use |
|---|---|---|
| **CLAUDE.md** | Context | Threat model baseline, forbidden patterns, review gates |
| **Hooks** | Enforcement | Secrets blocked before disk, SAST gates pre-commit |
| **Agents** | Isolated analysis | Read-only reviewers that cannot modify what they audit |
| **Skills** | Reusable runbooks | `/security-audit`, `/stride-threat-model`, `/red-team-simulation` |

The critical distinction: **CLAUDE.md is context** (Claude may follow it). **Hooks are architecture** (Claude cannot bypass them). Security-critical behavior belongs in hooks.

## The vulnerable app

VulnBank is a fictional banking REST API with intentional vulnerabilities across every OWASP Top 10 2021 category. Every vulnerability is annotated with its OWASP category and a realistic attack payload вЂ” the app is a teaching artifact as much as a scan target.

| Module | Vulnerabilities | OWASP |
|---|---|---|
| `main.py` | Hardcoded secrets, SSRF, debug mode, verbose errors | A02, A05, A10 |
| `auth.py` | MD5 passwords, JWT `none` algorithm, no rate limiting | A02, A07, A08 |
| `db.py` | SQL injection in every query, credentials in source | A03, A02 |
| `admin.py` | IDOR, broken access control, mass assignment | A01, A05 |
| `files.py` | Path traversal, unrestricted upload, directory listing | A01, A03 |

## Claude Code setup

### Skills вЂ” runbooks as slash commands

```yaml
# .claude/skills/security-audit/SKILL.md
---
name: security-audit
description: Perform a comprehensive security audit. Use when asked for a
  security review, vulnerability scan, or OWASP analysis.
allowed-tools: Read, Bash
---
```

### Agents вЂ” read-only, specialized, parallel

All agents are scoped to `tools: Read` вЂ” they cannot modify files. This is deliberate: audit agents should never have write access to what they're reviewing.

```bash
# Invoke two agents in parallel
Review auth.py using subagents вЂ” run appsec-reviewer and red-teamer in parallel
```

### Hooks вЂ” enforcement that cannot be bypassed

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Write|Edit|MultiEdit",
      "hooks": [{ "type": "command", "command": "python3 .claude/hooks/secrets-scanner.py" }]
    }]
  }
}
```

A hook that exits with code 2 blocks the tool call entirely. Claude receives the finding and corrects course вЂ” the bad file never touches disk.

## CI/CD pipeline

The GitHub Actions workflow runs on **manual trigger** with selectable parameters:

- **Target:** specific file or full `app/`
- **Review type:** `full-audit` | `sast-only` | `dependency-check`
- **Severity threshold:** `low` | `medium` | `high` | `critical`

Manual trigger is deliberate вЂ” full control during demos, no automatic scans that would normalize the vulnerabilities.

## Key design decisions

**Why hooks over CLAUDE.md for enforcement?**
CLAUDE.md is context вЂ” Claude may follow it. Hooks are architecture вЂ” Claude cannot bypass them. Security-critical behavior (no secrets in source, SAST gate on commit) belongs in hooks, not instructions.

**Why read-only agents?**
Agents that cannot write files cannot corrupt what they are analyzing. A security reviewer should analyze and report вЂ” never modify.

**Why open-source tooling?**
Anyone can clone and run this immediately вЂ” no accounts, no billing. Semgrep OSS, Bandit, pip-audit. The MCP architecture makes it trivial to swap in commercial tools (Snyk, Wiz) later.

**Why manual CI trigger?**
During a live demo, you trigger the scan intentionally and walk through the results. This shows the workflow, not just the output.
